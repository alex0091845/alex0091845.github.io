Chapter 1.1. The Concept of Objects

I know a lot of other tutorials start with printing out Hello World and things, and we'll do that, but I want to start with the core of Java--the concept of objects, because Java is all about objects and classes.

I think all of us are familiar with the concept of objects once I explain it. It's pretty intuitive!

The core idea is that there are classes of objects or instances. Not classes as in academic courses, but classifications of objects or instances.

For example, you might classify:

  • Your Toyota Corolla, your friend's Subaru Crosstrek, and your family Honda CRV into a Car class.
  • A pine tree, a maple tree, and a cedar tree into a Tree class.
  • You, a friend of yours, and a family member of yours into a Human class.

A more natural (though somewhat strange, still) way of saying these are:

  • My family's Honda CRV is a Car.
  • That pine tree is a Tree.
  • My friend is a Human.

Let's look at the object on the left. It has all the properties and actions as defined in the class. Let's name it human0. It has a name of "Tommy", an age of 22, and so on.

For the object on the right, let's name it human1. It has a name of "Isabelle," an age of 26, and so on.

When we tell human0 (aka Tommy) to take an action of drink() (yes, we tell objects to take actions), its hydrationLevel goes up. But notice Isabelle's hydrationLevel does not change.

So the actions usually change or alter the object's own states in some way.

To summarize:

  • Each individual object can act based on the action you tell them to do, and only that object acts according to the instructions we have given it.
  • Each individual object has their own states, and changing one object's states does not (usually) affect other objects' states.

You might think, what's the big deal?

Nothing! It's intuitive, right?

Just because you clap, doesn't mean your friend will clap.

Just because you're eating and getting less hungry, doesn't mean your friend is getting less hungry.

What's nice is that these states and actions are summarized and described in these things called classes.

You define states and actions for classes of objects. Then, each object acts similarly, but based on their own state.

You should know from the fact that I just came up with the arbitrary states and actions above, that:

  1. There's no way to represent all states and actions of a real-world object in software.
  2. What states and actions you include and define are up to you.

And guess what? Java already has a bunch of built-in classes--with their own states and actions--written and designed by other people. We can use them!

For example, there is a class for:

  • Taking in user input, called Scanner
  • Creating some colors so we can draw things, called Color
  • Working with files, called File

In Java, states are called fields. Actions are called methods. From now on, I'll use these terminology. Make sure to understand what I mean!

Uses of Classes and Objects

So… having said all this, what's the purpose of objects? Why do we need them?

There are many uses of software objects, mostly when you want to treat a bunch of related information (fields) and action (methods) together into one unit (object):

  • In a game where you have players and enemies, you can have a Player class and an Enemy class.
  • In a backend (data processing/management) software for hospitals, schools, banks, coffee shop, you can have a Patient class, a Student class, a BankAccount class, and a Customer and a PremiumCustomer class respectively.
  • ...and many more

Section Summary

Let's summarize what you learned:

  • A class:
    • Is defined by someone like you. People write their own classes all the time. Some people decide others can use the classes and the code they wrote.
    • Describes fields and methods for any object/instance of that class.
      • e.g., the Human class says that there's a field called name, and a method called drink() for any object of the Human class (any Human object)
  • An object:
    • Has its own individual field values and act on its own
      • e.g., the name field value for one Human object might be “Tommy.” The name field value for another might be “Isabelle.”
      • e.g., while both Tommy and Isabelle's hydrationLevel goes up by 1 every time they drink, Tommy and Isabelle's hydrationLevel's change individually.
  • Object-oriented programming:
    • Is defining classes to create objects, and then telling those objects to take certain actions!
